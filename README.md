# Домашнее задание к занятию «Микросервисы: подходы» - `Иншаков Владимир`

Вы работаете в крупной компании, которая строит систему на основе микросервисной архитектуры.
Вам как DevOps-специалисту необходимо выдвинуть предложение по организации инфраструктуры для разработки и эксплуатации.


## Задача 1: Обеспечить разработку

Предложите решение для обеспечения процесса разработки: хранение исходного кода, непрерывная интеграция и непрерывная поставка. 
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- облачная система;
- система контроля версий Git;
- репозиторий на каждый сервис;
- запуск сборки по событию из системы контроля версий;
- запуск сборки по кнопке с указанием параметров;
- возможность привязать настройки к каждой сборке;
- возможность создания шаблонов для различных конфигураций сборок;
- возможность безопасного хранения секретных данных (пароли, ключи доступа);
- несколько конфигураций для сборки из одного репозитория;
- кастомные шаги при сборке;
- собственные докер-образы для сборки проектов;
- возможность развернуть агентов сборки на собственных серверах;
- возможность параллельного запуска нескольких сборок;
- возможность параллельного запуска тестов.

Обоснуйте свой выбор.

## Задача 2: Логи

Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

- возможность дать ссылку на сохранённый поиск по записям логов.

## Задача 3: Мониторинг

Предложите решение для обеспечения сбора и анализа состояния хостов и сервисов в микросервисной архитектуре.
Решение может состоять из одного или нескольких программных продуктов и должно описывать способы и принципы их взаимодействия.

Решение должно соответствовать следующим требованиям:
- сбор метрик со всех хостов, обслуживающих систему;
- сбор метрик состояния ресурсов хостов: CPU, RAM, HDD, Network;
- сбор метрик потребляемых ресурсов для каждого сервиса: CPU, RAM, HDD, Network;
- сбор метрик, специфичных для каждого сервиса;
- пользовательский интерфейс с возможностью делать запросы и агрегировать информацию;
- пользовательский интерфейс с возможностью настраивать различные панели для отслеживания состояния системы.

## Решение 1: Обеспечение разработки

Решение для обеспечения разработки может быть реализовано при помощи нескольких сервисов:
- Система контроля версий может быть реализована на базе GitHub, GitLab, Bitbucket...
- Репозитории для хранения кода, артифактов, Docker образов могут быть реализованы на базе: Artifactory, Nexus, Harbor... 
- CI/CD инструменты могут быть реализованы при помощи GitLab CI/CD, Jenkins...

Если же такого рода "зоопарк технологий" не приемлем, то со всеми перечисленными требованиями может справиться правильно настроенный GitLab CI/CD:
- Может быть развернут в облаке.
- Содержит в себе систему контроля версий GIT.
- Есть возможность создавать репозитории на каждый сервис.
- Может выступать Docker registry для Docker образов.
- Полноценный функционал CI/CD.
- Безопасность хранения секретов обеспечивается встроенным механизмом CI/CD variables, а также поддержкой HashiCorp Vault.
- Возможность разворачивания агентов сборки на собственных серверах.
- Возможность различных сценариев по запуску тестов и сборок.

## Решение 2: Логи

Решением задачи по сбору и анализов логов будет применение стека ELK (ElasticSearch, Logstash, Kibana). 
- Для сбора логов на удаленных хостах понадобится устовка компонента Filebeat (именно он будет отслеживать изменения в файлах логов и направлять их в "центральное хранилище").
- Filebeat не накладывает особых требований на приложения и способен считывать логи с STDOUT.
- FileBeat обеспечит гарантированную доставку за счет ведения реестра записей.
- Logstash осуществит получение и нормализацию логов для удобства дальнейшей работы с ними.
- Elasticsearch позволит получить удобную фильтрацию и поиск необходимой информации в собранных логах.
- Kibana предоставит интерфейс с удобной отрисовкой для работы с логами, а также возможность дать ссылку на сохранённый поиск по записям логов.

## Решение 3: Мониторинг

Для решения данной задачи предлагаю использовать связку следующих технологий:

- Prometheus
- Node_exporter
- cAdvisor
- Alertmanager
- Grafana
- Отдельные exporters, характерные для сервисов (например, postgres_exporter, mysqld_exporter, redis_exporter)

Prometheus будет ответственен за сбор метрик со всех хостов, осбслуживающих систему.

Node_exporter будет собирать метрики хоста и предоставлять их по pull-модели в Prometheus.

cAdvisor необходим для сбора метрик работающих контейнеров.

Alertmanager задействуется для настройки отправки уведомлений по различным сценариям.

Grafana - компонент для наглядной отрисовки дашбордов, представления метрик, с возможностью делать запросы и агрегировать информацию. 

Настройка дашбордов в Grafana готова обеспечить контроль за состонием нашей системы.
